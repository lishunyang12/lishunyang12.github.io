在 NVIDIA Blackwell Ultra 上对 vLLM-Omni 的
ModelOpt FP8 与 NVFP4 扩散模型服务进行基准测试
Qwen-Image-2512 · 2× NVIDIA B300 (SM103) · 张量并行 = 2
vLLM-Omni 量化研究 · 2026
摘 要
我们在两块 NVIDIA B300 上、通过 vLLM-Omni 服务 Qwen-Image-2512 文生图扩散模型，评估已合入的 ModelOpt 量化路径 （#2913 ModelOpt FP8 auto-detect、#3570 ModelOpt mixed FP8/NVFP4）。在相同提示词、随机种子与采样步数下与 BF16 基线对比：FP8 与混合 FP8/NVFP4 将峰值运行显存降低 15.6%–16.9%、磁盘 checkpoint 体积缩减 35%–43%，同时保持输出保真度（在线 FP8：SSIM 0.9984）。量化配置在单流服务下更慢， 但在并发下吞吐超过 BF16，混合格式在并发 8 时达到 1.347 images/s（+18%）。我们定位并修复了 FP8 scaled-matrix-multiply 内核中的一处批处理缺陷， 并给出了 Blackwell Ultra（SM103）上的在线量化支持面——其上仅 FP8 可用。最后将这些结果与 NVIDIA 公开的 FLUX.2 NVFP4 数据相关联：二者在定性上一致，但不可直接比较。

vLLM-Omni 上的 ModelOpt FP8 与 NVFP4 扩散模型服务1  引言

1引言

训练后量化是在不重训的前提下降低大型扩散 Transformer 显存与延迟成本的主要手段。对文生图而言，工程上要回答的问题很具体：能回收多少显存、输出保真度是否保持， 以及在何种并发下低精度内核的算术收益开始压过其逐次调用的反量化开销。本文针对通过 vLLM-Omni 服务、运行在两块 NVIDIA B300（Blackwell Ultra， 计算能力 10.3 / SM103，张量并行为 2）上的扩散模型 Qwen-Image-2512 给出实证回答。

评估两类量化。离线量化加载预量化的 ModelOpt checkpoint，由 vLLM-Omni 自动识别，测试了全 FP8 与混合 FP8/NVFP4 两个 checkpoint；在线量化则在加载时 通过 --quantization <method> 对 BF16 checkpoint 施加量化。全部测量使用固定负载：1024x1024, 20 inference steps, seed 42, prompt='A ceramic teapot on a wooden table'。

2实验设置与样例输出

预量化 checkpoint 无需 --quantization 标志，由 checkpoint 配置选择路径，且已在启动时核验内核选择、无静默回退到 BF16。代码 1 给出四种服务命令。

# BF16 基线
vllm serve Qwen/Qwen-Image-2512 --omni --tensor-parallel-size 2
 
# 离线 ModelOpt FP8（预量化 checkpoint，自动识别）
vllm serve feizhai123/qwen-image-2512-modelopt-fp8-dynamic-all --omni \
  --tensor-parallel-size 2 --linear-backend cutlass --force-cutlass-fp8 --trust-remote-code
 
# 离线 混合 FP8/NVFP4
vllm serve feizhai123/qwen-image-2512-modelopt-mixed-fp8-sensitive-nvfp4-heavy --omni \
  --tensor-parallel-size 2 --linear-backend cutlass --force-cutlass-fp8 --trust-remote-code
 
# 在线 FP8（加载时对 BF16 权重做动态量化）
vllm serve Qwen/Qwen-Image-2512 --omni --tensor-parallel-size 2 --quantization fp8
代码 1. 四种配置在 vLLM-Omni 下的服务命令。

图 1 给出相同提示词、种子与 20 步调度下各精度的生成结果。三个量化输出与基线、彼此之间的校验和均不同，证实是真实的不同计算，而非缓存或静默回退。各配置的视觉保真度均得到保持， 差异仅限于细小的高频细节，且在混合 FP8/NVFP4 配置上最明显——与其更激进的权重压缩一致。

峰值显存降低 15.6%（离线 FP8）与 16.9%（混合）；checkpoint 缩减 35% 与 43%。在线 FP8 取得最高保真度（PSNR 39.3 dB，SSIM 0.9984）， 因为它在加载时对 BF16 权重做动态量化，而非依赖单独标定的产物。图 2、图 3 分别展示峰值显存与单请求延迟。

图 2. 各配置双卡峰值显存，越低越好。约 16% 的节省主要来自权重存储，而非混合格式层的精度。
图 3. 单请求平均延迟，越低越好。量化线性层在单请求时承担反量化开销——混合配置中的 NVFP4 最为显著（+52.8%）；在线 FP8 与 BF16 在噪声范围内持平。
单请求场景低估了量化的价值：单请求时 GEMM 规模小、反量化开销占主导。低精度的算术与带宽优势只有在并发饱和内核时才会兑现——见第 4 节。

vLLM-Omni 上的 ModelOpt FP8 与 NVFP4 扩散模型服务4  并发吞吐

4并发下的吞吐

图 4 给出并发 1、4、8 个在途请求时的持续吞吐（images/s），采用批处理服务（--step-execution --max-num-seqs 8，每点 16 个请求）。 可观察到明显的交叉：低并发下 BF16 最快，但随批次填满，量化格式的权重带宽节省压过其逐次开销。

图 4. 吞吐随并发变化，越高越好，标注并发 8 处的端点值。并发 8 时三种量化配置均超过 BF16（1.137）：混合 1.347（+18%）、在线 FP8 1.233（+8%）、离线 FP8 1.218（+7%）。混合配置从并发 1 到 8 扩展 2.34×。

该交叉是核心的运行结论：对延迟敏感的单流服务，本硬件上 BF16 仍更优；对吞吐导向的批处理服务，量化配置全面占优。这一“随机制依赖”与上游 Qwen3-Omni W4A4 工作（PR #4025）一致—— 量化优势同样在并发下、而非单流延迟中显现。

5FP8 路径的批处理缺陷及其修复

离线 ModelOpt FP8 路径在 --max-num-seqs 1 下服务正常，但一旦在 step execution 下批处理两条序列即确定性失败。编译后、序列打包的激活非连续， 而内核试图用 Tensor.view 重解释它——这在 torch.compile 的 fake-tensor 追踪下对非连续存储非法（代码 2）。

torch._dynamo.exc.TorchRuntimeError: RuntimeError when making fake tensor call
  File ".../layers/quantization/modelopt.py", line 611, in apply
  File ".../kernels/linear/scaled_mm/ScaledMMLinearKernel.py", line 134, in apply_weights
    x_2d = x.view(-1, x.shape[-1])
代码 2. 在 step execution 下批处理 FP8 路径时抛出的 fake-tensor 失败。

修复极小：将 Tensor.view 改为 Tensor.reshape——后者在布局允许时返回视图，仅在不可能时才拷贝（代码 3）。

-    x_2d = x.view(-1, x.shape[-1])
+    x_2d = x.reshape(-1, x.shape[-1])
代码 3. 对 ScaledMMLinearKernel.apply_weights 的一行修复（已作为 PR #4155 提交）。

修复后，离线 FP8 路径在所有测试并发下均无错批处理，并贡献了图 4 的吞吐。混合配置从未受影响，因其走 CUTLASS NVFP4 路径而非 scaled-matrix-multiply 内核。 在线 FP8 因共用同一路径，该修复同样适用。其长期归宿应是上游 vllm-project/vllm。

6Blackwell Ultra 上的在线量化支持

对四种在线方法分别以对应标志服务 BF16 checkpoint，结果见表 2：本硬件上仅 FP8 可用，其余三者因不同的内核级原因在加载时失败。

表 2. 2× B300（SM103）上的在线量化支持矩阵。
方法标志结果原因
FP8--quantization fp8可用—
INT8 (W8A8)--quantization int8加载失败CUTLASS scaled-mm 派发拒绝 SM103：“Int8 not supported on SM103.”
MXFP8 (W8A8)--quantization mxfp8加载失败NotImplementedError：仅支持 Ascend (NPU) / Intel (XPU)。
MXFP4 (W4A4)--quantization mxfp4加载失败NotImplementedError：仅 Ascend；CUDA 尚未实现。

值得注意的是，尽管文档将 Int8 列为“Blackwell SM100+”支持，INT8 W8A8 内核却拒绝 SM103，表明存在 Blackwell Ultra 特有的“文档—内核”落差。在本模型与硬件上， 在线 FP8 在各维度均持平或略胜离线 ModelOpt FP8——更低延迟（1.157 s 对 1.203 s）、相当的显存、更高保真度 （39.34 dB 对 29.07 dB）、并发 8 下持平或更高的吞吐（1.233 对 1.218 images/s）——且无需特制 checkpoint。 离线路径保留两项优势：更小的磁盘占用，以及用于更深显存压缩的混合 FP8/NVFP4 选项。

vLLM-Omni 上的 ModelOpt FP8 与 NVFP4 扩散模型服务7–8  关系与结论

7与 NVIDIA FLUX.2 NVFP4 结果的关系

NVIDIA 公开报告了 FLUX.2 [dev] 在 Blackwell 数据中心 GPU 上的 NVFP4 加速，给出相对 H200 基线 6.3× 单卡、10.2× 双卡的加速。这些数字与本研究不可直接比较， 也不应互相视为复现。模型不同（FLUX.2 [dev] 对 Qwen-Image-2512）；运行时不同（TensorRT-LLM visual_gen 原型对 vLLM-Omni）； 优化集不同（NVIDIA 的头条数字叠加了 NVFP4 + TeaCache 跳步 + CUDA Graphs + torch.compile，本研究仅隔离量化本身）；基线也不同（相对 H200 而非同机 B300 的 BF16）。

在可定性比较之处，二者一致：双方都发现低精度生成相对 BF16 近乎无损——NVIDIA 凭视觉、本研究凭量化指标（在线 FP8 SSIM 0.9984）。更实质地，本结果为 NVIDIA “针对 Blackwell Ultra GPU 的额外优化仍在进行中”这一说法提供了独立的内核级佐证：在 SM103 上，四种在线低精度方法有三种不可用，INT8 内核更直接拒绝该架构。 二者在单卡加速上存在分歧——NVIDIA 报告 NVFP4 单独约 2× 增益，而本研究观察到量化路径在单流下变慢——但该分歧源于不同的优化栈与测量机制，而非矛盾：NVIDIA 的单卡数字取自带跳步的融合 NVFP4 内核路径， 而本文隔离出的量化优势只在并发下显现。

8结论

对于通过 vLLM-Omni 服务、运行在 NVIDIA B300 上的 Qwen-Image-2512，FP8 与混合 FP8/NVFP4 量化在保持输出保真度的同时回收约 16% 运行显存、缩减 35%–43% checkpoint 体积， 并在并发饱和内核后超过 BF16 吞吐。本硬件上推荐以在线 FP8 为默认：它是 SM103 上唯一可用的在线低精度方法，相对 BF16 近乎无损，且在无需特制产物的情况下持平或胜过离线 FP8。 当磁盘占用或峰值显存是约束瓶颈时，离线混合 FP8/NVFP4 仍是合适之选。对 scaled-matrix-multiply 内核的一行修复（PR #4155）是批处理 FP8 服务的前提，建议上游采纳。
